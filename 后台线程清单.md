### 1. FBAlertsMonitor — 弹窗自动监控

- **源文件**: `WebDriverAgentLib/Utilities/FBAlertsMonitor.m`
- **队列标识**: `com.ecwda.alerts.monitor` (DISPATCH_QUEUE_SERIAL)
- **触发间隔**: 每 2 秒
- **当前状态**: ⛔ 已关闭（ECScriptParser.m 中 Session 创建时不再传入 defaultAlertAction）
- **触发条件**: 仅当 Session 创建时传入 `defaultAlertAction` 参数才会启动
- **XCTest IPC**: ⚠️ 高消耗 — 每次都查询 SpringBoard UI 树

#### 核心方法

| 方法 | 说明 |
|---|---|
| `- (void)enable` | 启动监控，设置 isMonitoring=YES 并调用 scheduleNextTick |
| `- (void)disable` | 停止监控 |
| `- (void)scheduleNextTick` | 每 2 秒在 monitorQueue 上执行一次弹窗检测 |

#### 调用链
```
FBSession initWithApplication:defaultAlertAction:  (FBSession.m:305)
  └→ [session enableAlertsMonitor]                  (FBSession.m:310)
      └→ [[FBAlertsMonitor alloc] init]             (FBSession.m:320)
          └→ [self.alertsMonitor enable]            (FBSession.m:322)
              └→ [self scheduleNextTick]            (FBAlertsMonitor.m:87)
                  └→ XCUIApplication.fb_systemApplication.fb_alertElement   (每2秒)
                      └→ [self.delegate didDetectAlert:alert]
                          └→ [self handleAlertSmart:alert]  (FBSession.m:80)
```

### 2. ECMAIN 保活探测定时器

- **源文件**: `WebDriverAgentLib/Categories/XCUIDevice+FBHealthCheck.m`
- **方法**: `+ (void)fb_checkEcmainHealth`
- **队列**: `dispatch_get_main_queue()` (NSTimer)
- **触发间隔**: 每 30 秒（首次延迟 30 秒）
- **启动方式**: `+load` 方法自动启动（进程加载时触发，无需手动调用）
- **XCTest IPC**: 🟡 中等 — 仅在重拉 ECMAIN 时触发

#### 核心方法

| 方法 | 说明 |
|---|---|
| `+ (void)load` | 类加载时自动注册 30 秒 NSTimer |
| `+ (void)fb_checkEcmainHealth` | 向 `http://127.0.0.1:8089/ping` 发送 GET 请求检测 ECMAIN 存活 |
| `+ (void)fb_relaunchEcmain` | 连续 3 次探测失败后，通过 FBUnattachedAppLauncher 重新拉起 ECMAIN (com.ecmain.app) |

#### 工作流程
```
+load (进程启动)
  └→ 延迟 30 秒后启动 NSTimer (每 30 秒)
      └→ fb_checkEcmainHealth
          ├→ HTTP GET http://127.0.0.1:8089/ping (超时 10s)
          ├→ 成功 → 重置失败计数器
          └→ 失败 → _ecmainFailureCount++
              └→ >= 3 次 → fb_relaunchEcmain
                  └→ [FBUnattachedAppLauncher launchAppInBackgroundWithBundleId:@"com.ecmain.app"]
                     (冷却 60 秒防重复拉起)
```

### 3. WDA 管道看门狗（Pipeline Watchdog）

- **源文件**: `WebDriverAgentLib/Categories/XCUIDevice+FBHealthCheck.m`
- **方法**: `+ (void)fb_startWDAPipelineWatchdog`
- **队列**: `dispatch_get_global_queue(QOS_CLASS_UTILITY)` (GCD dispatch_source_t)
- **触发间隔**: 每 20 秒（首次延迟 45 秒等待 HTTP 服务器完全初始化）
- **自检超时**: 8 秒
- **启动方式**: `+load` 方法自动启动
- **XCTest IPC**: 🟢 无 — 这是检测死锁的保护机制

#### 核心方法

| 方法 | 说明 |
|---|---|
| `+ (void)fb_startWDAPipelineWatchdog` | 创建 GCD Timer，每 20 秒触发自检 |
| `+ (void)fb_performWDAPipelineSelfCheck` | 向自身 `http://127.0.0.1:10088/health` 发 GET 请求（超时 8s），使用独立 ephemeral NSURLSession |

#### 工作原理
```
后台队列 → HTTP GET http://127.0.0.1:10088/health
                    ↓
          WDA主线程处理 /health 路由
                    ↓
    如果主线程被 XCTest IPC 死锁 → 请求超时
                    ↓
    连续 2 次超时 → exit(1) 自杀
                    ↓
    ECMAIN 探测到 10088 端口不通 → 重新拉起 ECWDA
```

### 4. MJPEG 截图推流服务

- **源文件**: `WebDriverAgentLib/Utilities/FBMjpegServer.m`
- **队列标识**: `JPEG Screenshots Provider Queue` (DISPATCH_QUEUE_SERIAL, QOS_CLASS_UTILITY)
- **触发间隔**: 根据 `mjpegServerFramerate` 配置（最高 60 FPS，即 ~16ms）
- **启动方式**: init 时自动启动循环调度
- **XCTest IPC**: 🔴 高消耗 — **但仅在有客户端连接时**

#### 核心方法

| 方法 | 说明 |
|---|---|
| `- (instancetype)init` | 初始化后台队列，立即调用 streamScreenshot |
| `- (void)streamScreenshot` | 核心截图循环。**无客户端时直接 return（空转零消耗）**；有客户端时调用 FBScreenshot 截图 |
| `- (void)scheduleNextScreenshotWithInterval:timeStarted:` | 计算下一帧延迟，通过 dispatch_after 调度 |
| `- (void)sendScreenshot:` | 将 JPEG 数据封装为 MJPEG chunk 推送给所有已连接客户端 |
| `- (void)didClientConnect:` | 新客户端连接回调 |
| `- (void)didClientSendData:` | 客户端首次发送数据后加入推流列表 |
| `- (void)didClientDisconnect:` | 客户端断开，从推流列表移除 |

#### 性能影响说明
```
listeningClients.count == 0 ?
  ├→ YES → 仅空转 dispatch_after 调度，无截图，无 XCTest IPC，零性能消耗
  └→ NO  → 每帧调用 [FBScreenshot takeInOriginalResolutionWithScreenID:...]
            走 XCTest IPC 通道截图，高 CPU + 高 IPC 负载
```

### 9. ECBackgroundManager — 心跳 + 8089 自检定时器

- **源文件**: `ECMAIN/ECMAIN/Core/ECBackgroundManager.m` (约 1605-1670 行)
- **队列**: `dispatch_get_global_queue(PRIORITY_HIGH)` (GCD dispatch_source_t)
- **触发间隔**: 首次 10 秒，之后每 60 秒
- **启动方式**: `startHeartbeat:` 方法调用（由 AppDelegate.didFinishLaunching 统一管控）
- **XCTest IPC**: 🟢 无

#### 核心方法

| 方法 | 说明 |
|---|---|
| `- (void)startHeartbeat:` | 创建 GCD Timer，首次 0.5 秒后立即发送一次心跳 |
| `- (void)sendHeartbeat:` | 向云控制中心发送设备心跳数据（UDID/状态/VPN 等） |

#### 附带的 8089 自检（嵌入在心跳定时器的回调中）
```
每 60 秒心跳触发时同步执行：
  └→ HTTP GET http://127.0.0.1:8089/ping (超时 3s)
      ├→ 成功 → 重置 _port8089FailCount
      └→ 失败 → _port8089FailCount++
          └→ >= 2 次 → 重启 ECWebServer 8089 端口服务
```

## 三、XCTest IPC 风险等级总览

| 风险 | 线程 | 说明 | 运行状态 |
|---|---|---|---|
| 🔴 高 | MJPEG Server | 有客户端连接时高频截图（每帧一次 IPC）。**无客户端时零消耗** | ✅ 运行中（空转） |
| 🔴 高 | FBAlertsMonitor | 每 2 秒查询 SpringBoard UI 树 | ⛔ 已关闭 |
| 🟡 中 | Health Check | 重拉 ECMAIN 时调用 XCTest IPC | ✅ 运行中 |
| 🟡 中 | FBScriptEngine | 脚本执行时 OCR/截图/触摸走 XCTest IPC | ✅ 按需运行 |
| 🟢 无 | 其余 14+ 个线程 | 纯网络/音频/文件 I/O | ✅ 运行中 |
