# ECMAIN 应用安装功能详解

ECMAIN 作为一款高度定制化的 iOS 应用管理与自动化框架，其内置了多种应用（IPA）安装方式。这是为了应对不同来源的 App（越狱商店、官方砸壳、带壳原包等）以及不同的应用使用场景（如多开、注入伪装等）而专门设计的。

以下是应用列表中支持的所有安装方式的功能剖析与代码执行逻辑分析：

---

## 1. 安装User (普通用户安装)
*   **功能描述**：使用常规的 TrollStore 机制将 IPA 解压并安装到系统，同时在 LaunchServices 中将其注册为 **User (用户级)** 应用。
*   **适用场景**：安装绝大部分普通的第三方应用、去广告版应用等。
*   **状态分析**：**正常**。代码调用了 `installIPA:registrationType:@"User"`，依赖 TrollStore 底层标准安装流程，运行稳定。

## 2. 安装System (系统级保护安装)
*   **功能描述**：同样基于 TrollStore 机制安装，但强制将应用的注册类型指定为 **System (系统级)**。
*   **适用场景**：这是针对 iOS 14+ 及特定环境下，User 级应用在设备重启后容易图标消失、LaunchServices 丢失配置这一 Bug 所做的**强力防丢措施**。ECMAIN 核心管理的辅助应用及自动化脚本目标 App 必须使用此方式，以确保其常驻生存。
*   **状态分析**：**正常**。也是 ECMAIN 环境中最推荐的默认普通安装方式。

## 3. 原包安装 (Original - 官方 LSWorkspace API 安装)
*   **功能描述**：不同于 TrollStore 的直接物理路径解压与 CoreTrust 漏洞强行注册，此模式调用了 Apple 原生的 `[LSApplicationWorkspace installApplication:withOptions:]` API 进行安装。
*   **适用场景**：
    *   某些包含极其复杂的 Frameworks 依赖或对安装路径权限要求极度严格的大型 App。
    *   在 TrollStore 暴力解压法（Method 1）导致 App 闪退或权限异常时，作为一种兼容性退回手段 (Fallback)。
*   **状态分析**：**正常**。内部通过 `installOriginalIPA:` 触发 `_performInstallIPA`，Method=0，走系统标准框架解析。

## 4. 注入并安装 (System) (自动化伪装库注入安装)
*   **功能描述**：这是 ECMAIN 的**核心独家特性**。点击后会先拦截安装流程，将 IPA 解压到临时目录，随后弹出高级配置菜单，允许用户分配**分身编号**，并填写**设备指纹伪装数据**。接着系统会自动将伪装 dylib（动态链接库）注入到 App 的二进制文件中，再进行重签名与安装。
*   **适用场景**：专为 TikTok 等需要防封控、改机、伪装硬件信息（修改 IMEI、基带、坐标等）的黑产或跨境矩阵业务开发。
*   **状态分析**：**正常**。代码中完整实现了从解压、读取 `Info.plist`、生成 `device.plist`（存入 Payload）、注入 dylib 到最后封装的闭环。

## 5. 分身安装 (System) (多开安装)
*   **功能描述**：在安装前动态修改应用的 `CFBundleIdentifier`（Bundle ID）和 `CFBundleDisplayName`（应用名称），然后以系统级权限安装。
*   **适用场景**：需要在同一台手机上安装同一个 App 的多个独立运行版本（如微信多开、TikTok 矩阵多开），使它们沙盒相互隔离。
*   **状态分析**：**正常**。调用 `showCloneInstallAlertForPath:skipSigning:NO` 触发，配合底层签名修正机制可完美实现多开。

## 6. 🔐 加密安装 (User) (带壳 DRM 安装)
*   **功能描述**：强制**跳过所有的 `ldid` 重签名和 CoreTrust 签发步骤**（`skipSigning:YES`），直接将包含原始加密块（Cryptid=1）和 `SC_Info`（FairPlay DRM 数据）的文件放入系统。
*   **适用场景**：用于安装从别人手机或直接从 App Store 拦截导出的**未砸壳（Encrypted）原始 IPA 文件**。如果对此类文件进行常规重签名，签名工具会破坏原有的 DRM 结构导致应用闪退。跳过签名后，只要手机登录了下载该 App 的有效 Apple ID，应用就能正常运行。
*   **状态分析**：**正常**。调用了 `handleAppInstallFromFile` 并将 `skipSigning` 参数设为 `YES`。

---

### 总结：为什么需要这么多种安装方式？

由于 ECMAIN 主要面向**设备农场、矩阵号自动化、出海应用管理**等复杂商业场景，普通的傻瓜式安装根本无法满足需求。
1. **防图标消失**催生了 `System` 安装。
2. **账号矩阵隔离**催生了 `分身` 安装。
3. **风控规避与改机**催生了 `注入` 安装。
4. **App 兼容性适配**催生了 `原包` 与 `加密` 安装。

目前以上接口在底层源码中均有严格的逻辑分支支撑，并且状态均正常运作。
