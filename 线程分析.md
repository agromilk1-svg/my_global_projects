# ECMAIN 线程架构与优化方案

> 基于 ECMAIN 全量源码深度审计，以下按启动顺序梳理所有线程/队列的生命周期与资源消耗，
> 并给出不影响功能的安全优化建议。

---

## 一、线程全景地图

ECMAIN 启动后，`AppDelegate.didFinishLaunchingWithOptions` 依次拉起以下子系统，
每个子系统背后都携带一个或多个线程/队列：

| # | 启动顺序 | 子系统 | 线程类型 | 周期 | 资源等级 |
|---|---------|--------|---------|------|---------|
| 1 | `ECNetworkManager` | 旧版网络轮询 | NSTimer (Main RunLoop) | 5s | ⚪ 极低（空实现） |
| 2 | `ECWebServer` | HTTP 文件/API 服务 | NSThread (serverThread) | 常驻 | 🟢 低 |
| 3 | `ECKeepAlive` | 静音音频后台保活 | GCD Timer (HIGH) | 120s 自检 | 🟢 低 |
| 4 | `ECBackgroundManager` | 心跳 + WS 隧道 + MJPEG | GCD Timer (HIGH) | 60s | 🟡 中 |
| 5 | `ECTaskPollManager` | 云端自动脚本轮询 | GCD Timer (Serial) | 60s | 🟡 中 |
| 6 | `ECOneshotTaskManager` | 一次性任务轮询 | GCD Timer (Serial) | 90s | ⚪ 极低 |
| 7 | `ECLogManager` | 日志持久化 | GCD Serial Queue | 按需 | ⚪ 极低 |
| 8 | `ECConfigManager` | 配置读写 | GCD Serial Queue | 按需 | ⚪ 极低 |
| 9 | `ECScriptParser` | 自动化脚本执行器 | Global Concurrent | 按需 | 🔴 高 |
| 10 | Tunnel 隧道 (VPN) | 封包 TCP/UDP 转发 | 多条 GCD Serial | 常驻 | 🔴 高 |
| 11 | AppDelegate 后台刷新 | spawnRoot 刷新注册 | Global Queue (一次性) | 启动时 | ⚪ 极低 |
| 12 | AppDelegate 评论同步 | HTTP 批量拉取 | Global Queue LOW (一次性) | 启动时 | ⚪ 极低 |

## 二、逐线程详细分析

### 1. `ECNetworkManager` — 🗑️ 废弃待清理

**文件**: `Network/ECNetworkManager.m`

```objc
// pollTask 方法体为空（TODO 注释）
- (void)pollTask {
  // TODO: Implement actual URL fetch
}
```

- 每 **5 秒**在主线程 RunLoop 触发一次 `pollTask`，但方法体是**空的**。
- 功能已完全被 `ECTaskPollManager` (60s GCD Timer) 取代。
- 主线程上这个空 Timer 虽不消耗 CPU，但**占用 RunLoop 唤醒次数**，
  对 iOS 的电量管理有微弱负面影响。

**优化方案 ✅ 安全**:
> 删除 `AppDelegate.m` 第 30 行的 `startPolling` 调用。
> 在 `ECNetworkManager.m` 中将 `pollTimer` 初始化注释掉或将 Timer 间隔改为 0（不创建）。
> 不影响任何功能——因为 `pollTask` 本身什么都没做。

---

### 2. `ECWebServer` — 🟢 健康

**文件**: `Network/ECWebServer.m`

- 创建了一条专属 `NSThread` (`serverThread`) 运行 RunLoop，承载 HTTP 监听。
- 空闲时处于 `mach_msg_trap` 休眠，CPU 消耗为零。
- 文件传输时消耗短暂升高但属于正常预期。

**无需优化。**

---

### 3. `ECKeepAlive` — 🟢 健康，可微调

**文件**: `ECMAIN/Core/ECKeepAlive.m`

- 静音 `AVAudioPlayer` 无限循环 + GCD Timer 每 120s 自检。
- 自检逻辑极轻（判断 `isPlaying` 状态），但自检日志输出频繁：

```objc
// 每 120 秒输出一次
NSLog(@"[ECKeepAlive] ✅ [定时自检] 静音播放正常运行中");
```

**优化方案 ✅ 安全**:
> 1. 将正常状态下的日志去掉或降级为 `#ifdef DEBUG` 保护，减少主线程 NSLog 序列化开销。
> 2. 保留异常状态日志（播放停止时的恢复逻辑不变）。

---

### 4. `ECBackgroundManager` 心跳定时器 — 🟡 需要优化

**文件**: `ECMAIN/Core/ECBackgroundManager.m` (L1886-1975)

每 **60 秒**在 `DISPATCH_QUEUE_PRIORITY_HIGH` 上触发，单次触发内执行以下操作：

| 操作 | 开销 |
|------|------|
| `sendHeartbeat` (HTTP POST) | 🟢 网络 IO |
| MJPEG 空闲看门狗检查 | ⚪ 纯计算 |
| 8089 端口 HTTP 探活 | 🟡 网络 IO |
| ECWDA 10088 端口探活 (Heartbeat 分类中) | 🟡 网络 IO |

**问题点**:
1. **三次独立 HTTP 请求** (心跳 + 8089 + 10088) 在同一次定时器回调中触发，
   若网络抖动（飞行模式切换后恢复），三者同时超时会阻塞 `DISPATCH_QUEUE_PRIORITY_HIGH`
   线程池中的线程长达 `3 × 3s = 9s`。
2. 心跳定时器运行在 `DISPATCH_QUEUE_PRIORITY_HIGH` 优先级全局队列上，
   这与系统动画、UI 响应竞争 CPU 时间片。

**优化方案 ✅ 安全**:
> 1. 将心跳 GCD Timer 的队列从 `DISPATCH_QUEUE_PRIORITY_HIGH` 降级为
>    `DISPATCH_QUEUE_PRIORITY_DEFAULT`，心跳本身不是实时操作，
>    没有理由抢占 UI 线程优先级。
> 2. 8089 与 10088 的探活请求可以**交错执行**（奇数次心跳探 8089，偶数次探 10088），
>    避免同一回合三路 HTTP 并发。
>    这不会降低发现问题的速度（从 60s 变为 120s 仍然足够快）。

---

### 5. `ECTaskPollManager` — 🟡 日志可优化

**文件**: `ECMAIN/Core/ECTaskPollManager.m`

- 每 60s 向云控服务器 GET 拉取脚本列表，已有完善的指数退避和飞行模式跳过。
- 架构上没有问题。

**问题点**:
- `NSDateFormatter` 每次调用 `todayDateString` / `currentDateTimeString` 时都**重新分配**。
  `NSDateFormatter` 的创建开销较高（底层依赖 ICU 库初始化）。

**优化方案 ✅ 安全**:
> 将 `todayDateString` 和 `currentDateTimeString` 中的
> `NSDateFormatter` 抽为类属性或静态局部变量复用，避免每次 alloc/init。

---

### 6. `ECOneshotTaskManager` — ⚪ 无需优化

**文件**: `ECMAIN/Core/ECOneshotTaskManager.m`

- 90s 轮询间隔，逻辑极简，仅在有任务时才触发执行。
- 已有完善的防重入保护 (`isExecuting`)。

**无需优化。**

---

### 7. `ECLogManager` — 🟢 可微调

**文件**: `ECMAIN/Core/ECLogManager.m`

- 串行队列写入磁盘，架构合理。

**问题点**:
- 每条日志调用了 `NSLog(@"%@", message)`（第 87 行），**相当于双写**——
  既写磁盘文件，又通过 `os_log` 写系统日志。
  在高频日志场景下（如脚本执行中循环打印），`NSLog` 的序列化锁会成为瓶颈。

**优化方案 ✅ 安全**:
> 在 Release 模式下去掉 `log:` 方法中的 `NSLog` 输出（保留文件写入即可），
> 或将其改为条件编译：
> ```objc
> #ifdef DEBUG
>   NSLog(@"%@", message);
> #endif
> ```
> 这样在设备上运行时只走文件写入，减少一半的 IO 序列化开销。

---

### 8. `ECScriptParser` 脚本执行 — 🔴 最大瓶颈

**文件**: `ECMAIN/Core/ECScriptParser.m`

脚本执行时在 Global Concurrent Queue 上阻塞运行，主要开销来源：

| 操作 | 频次 | 开销 |
|------|------|------|
| `[NSThread sleepForTimeInterval:]` | 每步 0.1-3s | 🟡 阻塞线程 |
| `spawnRoot(...)` | 每次 App 操作 | 🔴 Fork 子进程 |
| HTTP 请求 (脚本内网络调用) | 按需 | 🟡 IO |

**问题点**:
1. `spawnRoot` 内部通过 `posix_spawn` 孵化独立进程执行 Shell 操作。
   系统级 Fork 开销极高（内存页表复制、进程描述符分配），
   当脚本中连续执行多次 `killAllApps` / `wipe-app` 时，CPU 会瞬间飙至 100%。
2. `[NSThread sleepForTimeInterval:]` 虽然不消耗 CPU，
   但会**长时间占用 GCD 全局线程池中的线程**，
   如果多个脚本/任务并行（虽然目前有 `isExecuting` 守卫），
   全局队列的线程数量会被耗尽。

**优化方案 ✅ 安全（不影响功能）**:
> 1. **合并批量 spawnRoot 调用**：对于如 `chmod-file`、`copy-file` 等连续操作，
>    可以在 RootHelper 中新增一个 `batch` 子命令，
>    一次传入多个操作的 JSON 数组，在一个子进程内循环执行，
>    避免反复 Fork。这是**投入产出比最高**的优化。
> 2. 将脚本执行从 Global Queue 迁移到**专属串行队列**：
>    ```objc
>    static dispatch_queue_t _scriptQueue;
>    _scriptQueue = dispatch_queue_create("com.ecmain.script", DISPATCH_QUEUE_SERIAL);
>    ```
>    好处：明确隔离脚本执行的资源消耗，不再影响全局并发池。

### 9. Tunnel 隧道相关队列 — 🔴 常驻高消耗

**文件**: `Tunnel/ECTun2Proxy.m`, `Tunnel/ECHevTunnel.m`, `Tunnel/ECUDPBridge.m`

| 队列 | 类型 | 用途 |
|------|------|------|
| `com.ecmain.tun2proxy.run` | Serial | tun2proxy 主循环 |
| `com.ecmain.tun2proxy.read` | Serial | 读取代理响应 |
| `com.ecmain.hevtunnel` | Serial | Hev 隧道封包 |
| `com.ecmain.udpbridge` | Serial | UDP 桥接 |

- 当 VPN 隧道处于激活状态时，**所有 App 的网络流量**都经由这些队列中转。
- CPU 消耗与网络流量成正比，高带宽（视频流/大文件下载）时持续 20-40%。

**优化方案 ✅ 安全（不影响功能）**:
> 1. 在 `PacketTunnelProvider` 中引入**流量闲置检测**：
>    如果 30 秒内无数据包流过，将 dispatch_source 暂停，
>    有新数据包时自动恢复。避免空转消耗。
> 2. `ECUDPBridge` 中的 `dispatch_source_create(DISPATCH_SOURCE_TYPE_READ, ...)`
>    已经是事件驱动模型（好的），但 `ECHevTunnel` 中主循环可能存在轮询空转——
>    确认 `hev_socks5_tunnel_run` 是否内部阻塞等待，还是忙等待。

---

### 10. `ECBackgroundManager` WS Ping Timer — 🟡 可优化

**文件**: `ECMAIN/Core/ECBackgroundManager.m` (L1052-1083)

- 每 15 秒通过 `NSTimer` 发送 WebSocket Ping。
- 此 Timer 运行在主 RunLoop 上。

**问题点**:
- 15 秒一次的 Ping 主要是为了防止 iOS 回收空闲 TCP 连接（60 秒阈值）。
- 但 15 秒对于 60 秒阈值来说过于频繁，**30 秒完全足够**。

**优化方案 ✅ 安全**:
> 将 `scheduledTimerWithTimeInterval:15.0` 改为 `30.0`，
> 仍然远小于 iOS 的 60 秒空闲回收阈值，但主线程唤醒频率直接减半。

---

### 11. `ECKeepAlive` 自检定时器 + `ECBackgroundManager` 心跳定时器 — 职责重叠

两个独立的 GCD Timer 都运行在 `DISPATCH_QUEUE_PRIORITY_HIGH`：

| Timer | 间隔 | 队列 | 职责 |
|-------|------|------|------|
| `ECKeepAlive._selfCheckGCDTimer` | 120s | Global HIGH | 检查音频播放是否存活 |
| `ECBackgroundManager._heartbeatGCDTimer` | 60s | Global HIGH | 心跳 + 端口探活 + 推流看门狗 |

**优化方案 ✅ 安全**:
> 将 `ECKeepAlive` 的自检合并到 `ECBackgroundManager` 心跳回调中
> （在 `sendHeartbeat` 旁加一行 `[[ECKeepAlive sharedInstance] selfCheck]`）。
> 减少一个独立的 GCD Timer 占用，同时保持 120s 的自检频率
>（每隔一次心跳触发一次，60s × 2 = 120s）。

## 三、优先级排序总结

按**投入产出比**排序，从最容易做且收益最大的开始：

| 优先级 | 优化项 | 预估工作量 | 节省资源 |
|--------|--------|-----------|---------|
| ⭐⭐⭐ | 删除 `ECNetworkManager.startPolling` 空轮询 | 1 行代码 | 消灭无用 5s Timer |
| ⭐⭐⭐ | 心跳 Timer 从 HIGH 降至 DEFAULT 优先级 | 1 行代码 | 不再抢占 UI 线程 |
| ⭐⭐⭐ | WS Ping 间隔从 15s 改为 30s | 1 行代码 | 主线程唤醒减半 |
| ⭐⭐ | Release 模式去掉 ECLogManager 中的 NSLog 双写 | 3 行代码 | 高频日志场景性能翻倍 |
| ⭐⭐ | ECKeepAlive 自检合并到心跳定时器 | 10 行代码 | 减少 1 个 GCD Timer |
| ⭐⭐ | 8089/10088 端口探活交错执行 | 15 行代码 | 避免 3 路 HTTP 并发超时 |
| ⭐⭐ | ECTaskPollManager 复用 NSDateFormatter | 5 行代码 | 减少频繁对象分配 |
| ⭐ | 脚本执行迁移到专属串行队列 | 10 行代码 | 隔离 GCD 池污染 |
| ⭐ | RootHelper 支持 batch 命令减少 Fork | 50 行代码 | 批量操作 CPU 降 50%+ |

## 四、不建议动的部分

以下模块虽然消耗资源但**架构合理，不应改动**：

1. **`ECKeepAlive` 静音播放本身** — 这是后台保活的核心生命线，`volume=0.01` 已经是最优解。
2. **`registerInfiniteBackgroundTask` 递归续命** — iOS 后台任务的标准防杀策略。
3. **ECWDA 看门狗的 5 次阈值和 120s 冷却** — 上轮优化已经调参到位。
4. **MJPEG 180s 空闲超时** — 是推流安全阀的最后防线，不能缩短。
